## 整体架构与设计理念

### 核心问题与解决方案

旧版工作流（v1.1.0）的主要缺陷：

| 缺陷 | 影响 | v1.2.0 解决方案 |
|------|------|-----------------|
| 全局 DICT 检索（一次性检索所有字典） | 返回大量无关字典内容，干扰 SQL 生成 | **两阶段设计**：先确定表，再按表检索 DICT |
| SQL 规划前缺少明确表选择步骤 | 模型容易幻觉出不存在的表名 | 新增 `Agent:SmartTablesPick` 作为表规划专家 |
| 综合分析缺少字典值映射 | 编码值直接展示，可读性差 | 将按表 DICT 传递给综合分析节点 |

### 两阶段设计理念

```
第一阶段：先确定"查哪些表"
  ↓ 基于 DB Description + DDL + Q-SQL + CrossTable 决策
  ↓ 输出：{darm_tables, datarms_tables, all_tables}

第二阶段：再检索"这些表的字典值"
  ↓ 对 all_tables 中每张表单独检索 DICT 知识库
  ↓ 输出：每张表的字段编码映射

第三阶段：携带精准 DICT 生成 SQL
  ↓ 基于表规划 + 精准字典 + DDL + DB + Q-SQL 生成 SQL 数组
  ↓ 字典值准确，WHERE 条件不再猜测编码值
```

## 各阶段说明

### 阶段一：问题理解

**节点**：`Agent:BrightMindsThink`

**作用**：将用户原始问题改写成清晰、完整的检索查询，支持多轮对话和简称补全。

**关键行为**：
1. 调用 `Tool:AliasLookupDeep`（简称映射工具）检查是否有需要展开的简称
2. 结合对话历史补全省略的信息（如"上面提到的那条路"）
3. 输出一个完整、独立的问题句子

**示例**：
- 用户输入："新台上的摄像枪有多少？"
- 改写输出："新台高速上摄像枪的数量是多少？"

---

### 阶段二：知识库检索（并行）

**4 个检索节点同时执行**：

| 节点 | 知识库 | 作用 |
|------|--------|------|
| `Retrieval:DeepFieldsSeek` | DB Description | 表字段的中文含义描述 |
| `Retrieval:WideTablesFind` | DDL | 建表语句（字段名、类型约束） |
| `Retrieval:SharpQueriesMatch` | Q-SQL | 历史Q-SQL范例 |
| `Retrieval:CrossTableMapping` | 跨表映射 | 同一实体在多系统的表对应关系 |

**并行效率**：4 个节点同时执行，不互相等待，提升整体响应速度。

**检索结果流向**：
- → `Agent:SmartTablesPick`（用于表规划决策）
- → `Agent:ClearPathsPlan`（用于SQL生成上下文）

---

### 阶段三：表规划

**节点**：`Agent:SmartTablesPick`

**这是 v1.2.0 新增的关键节点！**

**作用**：在生成SQL之前，先明确"应该查哪些表"，防止模型幻觉。

**输入**：4个知识库检索结果 + 用户问题

**输出（结构化JSON）**：
```json
{
  "reasoning": "分析过程：用户问题涉及摄像枪统计，需要查询机电系统gjdt_jd_base_camera、路运系统ldly_base_d_camera、原始表jtsjzy_camera...",
  "darm_tables": [],
  "datarms_tables": ["gjdt_jd_base_camera", "ldly_base_d_camera", "jtsjzy_camera"],
  "all_tables": ["gjdt_jd_base_camera", "ldly_base_d_camera", "jtsjzy_camera"]
}
```

**深度思考原则**：
- 广度覆盖：含"所有系统"时，考虑所有相关系统的同类表
- 多系统关联：同一业务实体在多系统的表都要纳入
- 只能选择在 DDL 中有证据的表（防止幻觉）

---

### 阶段四：按表 DICT 检索

**节点**：`ListOperations:DeduplicateTables` + `Iteration:DictPerTable`

**作用**：对表规划中的每张表，单独检索其字典值（字段编码映射）。

**流程**：
1. `ListOperations:DeduplicateTables`：对 `all_tables` 数组去重
2. `Iteration:DictPerTable`：迭代每张表名
   - `IterationItem:DictTableStart`：获取当前表名（迭代变量）
   - `Retrieval:TableDictFetch`：以表名为查询词，检索 DICT 知识库
3. 迭代完成后输出 `table_dicts[]`：每张表对应的字典内容数组

**输出示例**：
```
table_dicts = [
  "#gjdt_jd_base_camera：\n-road_no：11:佛开高速, 12:开阳高速...\n-is_active：0:停用, 1:启用",
  "#ldly_base_d_camera：\n-status：1:停用, 2:在用...",
  ...
]
```

**优势**：精准检索，只获取实际需要查询的表的字典，避免无关字典干扰。

---

### 阶段五：SQL 规划

**节点**：`Agent:ClearPathsPlan`

**作用**：基于表规划方案 + 精准字典 + 完整检索上下文，生成多库SQL数组。

**输入（7项）**：
1. 用户改写问题（`Agent:BrightMindsThink@content`）
2. 表规划方案（`Agent:SmartTablesPick@content`）
3. 表字段描述（`Retrieval:DeepFieldsSeek@formalized_content`）
4. 建表语句（`Retrieval:WideTablesFind@formalized_content`）
5. Q-SQL范例（`Retrieval:SharpQueriesMatch@formalized_content`）
6. 跨表映射（`Retrieval:CrossTableMapping@formalized_content`）
7. 按表字典值（`Iteration:DictPerTable@table_dicts`）← **精准！**

**输出（结构化）**：
```json
{
  "analysis_plan": "统计摄像枪，涉及3个系统...",
  "darm_sqls": [],
  "datarms_sqls": [
    "SELECT COUNT(*) AS '机电摄像枪数量' FROM gjdt_jd_base_camera WHERE commonDelStatus='0';",
    "SELECT COUNT(*) AS '路运摄像枪数量' FROM ldly_base_d_camera WHERE commonDelStatus='0';",
    "SELECT COUNT(*) AS '原始摄像枪数量' FROM jtsjzy_camera;"
  ]
}
```

**SQL生成策略（深度思考）**：
- 同一类数据在多个系统时，每个系统各生成1条SQL
- 支持 darm + datarms_ods 跨库分析
- 使用精准字典值（如 `is_active=1` 表示启用）

---

### 阶段六：SQL 执行（并行）

**两个并行迭代**：

#### `Iteration:StrongDarmSweep`（darm库迭代）
- 迭代项：`Agent:ClearPathsPlan@structured.darm_sqls`
- 内部节点：`ExeSQL:SafeDarmRun` → `Agent:WiseDarmRead`
- `ExeSQL:SafeDarmRun`：连接 darm 数据库执行SQL
- `Agent:WiseDarmRead`：提炼SQL执行结果（data_summary, key_findings, data_table, record_count）

#### `Iteration:DeepOdsSweep`（datarms_ods库迭代）
- 迭代项：`Agent:ClearPathsPlan@structured.datarms_sqls`
- 内部节点：`ExeSQL:SafeOdsRun` → `Agent:WiseOdsRead`
- `ExeSQL:SafeOdsRun`：连接 datarms_ods 数据库执行SQL
- `Agent:WiseOdsRead`：提炼SQL执行结果

**注意**：当SQL数组为空时，RAGflow的迭代节点会自动跳过，无需额外判断。

---

### 阶段七：汇总与综合分析

**节点链**：`VariableAggregator:MergeAllResults` → `Switch:FairResultsCheck` → `Agent:GrandMindsSynth`

#### `VariableAggregator:MergeAllResults`
合并两个迭代的结果为 `all_result` 数组：
```
all_result = [
  ...Iteration:StrongDarmSweep@result,    // darm库查询提炼结果
  ...Iteration:DeepOdsSweep@result1       // ods库查询提炼结果
]
```

#### `Switch:FairResultsCheck`
- `all_result` 为空 → `Message:CalmWordsNote`（友好提示）
- `all_result` 非空 → `Agent:GrandMindsSynth`（深度分析）

#### `Agent:GrandMindsSynth`（综合分析）
**输入**：
1. 用户改写问题
2. 分析方案（`Agent:ClearPathsPlan@structured.analysis_plan`）
3. 汇总查询结果（`VariableAggregator:MergeAllResults@all_result`）
4. 字典值参考（`Iteration:DictPerTable@table_dicts`）← **用于编码值转中文！**

**输出报告结构**：
1. **数据概览** - 各系统数据总体情况
2. **详细数据** - Markdown 表格展示
3. **深度分析** - 跨系统对比、趋势、关联
4. **关键发现** - 加粗标注重要发现
5. **总结** - 基于数据的综合结论

## 常见问题排查

### Q1：表规划选择了错误的表怎么办？

**原因**：Q-SQL知识库或DB Description知识库的召回内容不准确。

**解决**：
1. 检查 `Retrieval:WideTablesFind` 和 `Retrieval:DeepFieldsSeek` 的 `top_n` 是否足够（建议12-15）
2. 检查知识库的解析配置（建议使用 chunk_size=512, overlap=100）

### Q2：DICT检索没有返回结果怎么办？

**原因**：DICT知识库中没有对应表名的内容，或相似度阈值太高。

**解决**：
1. 检查 `Retrieval:TableDictFetch` 的 `similarity_threshold`（建议0.15-0.2）
2. 确认 DICT 知识库中表名格式与数据库表名一致

### Q3：SQL执行出现"字段不存在"错误？

**原因**：模型幻觉出了DDL中不存在的字段。

**解决**：
1. 增加 `Retrieval:WideTablesFind` 的 `top_n`（从12增加到15-20）
2. 在 `Agent:ClearPathsPlan` 的系统提示中进一步强调"只用DDL中真实存在的字段"

### Q4：深度思考结果太浅（SQL太少）？

**原因**：`Agent:SmartTablesPick` 选表不够广。

**解决**：
1. 增加 `Retrieval:CrossTableMapping` 的 `top_n`（从5增加到8-10）
2. 在用户问题中加入"全面"、"所有系统"等词触发广度覆盖

### Q5：综合分析报告中字段值仍然是编码数字？

**原因**：`Iteration:DictPerTable@table_dicts` 没有对应表的字典内容。

**解决**：
1. 检查DICT知识库中是否有该表的字典信息
2. 调低 `Retrieval:TableDictFetch` 的 `similarity_threshold`
