# 新博数据管家——问数深度思考 v1.3.0 工作流说明文档

## 版本变更说明

| 版本 | 核心变更 |
|------|----------|
| v1.0.0 | 基础深度思考：改写问题 → 4路检索 → 多SQL生成 → 并行执行 → 综合分析 |
| v1.1.0 | 新增工具调用节点（Tool:AliasLookupDeep）、字典检索节点（Retrieval:GetDictValues） |
| v1.2.0 | 尝试加入按表字典检索迭代（存在导入问题） |
| **v1.3.0** | **两阶段表规划+按表字典迭代：优先确定目标表，再按表精准检索字典，最后生成SQL，支持两库并行执行** |

---

## 架构总览

```
begin
  └─► 问题改写智能体（Agent:BrightMindsThink）
        ├─► DB描述检索（Retrieval:DeepFieldsSeek）     ─┐
        ├─► DDL检索（Retrieval:WideTablesFind）          ├─► 深度表规划智能体（Agent:SmartTablesPick）
        ├─► Q-SQL检索（Retrieval:SharpQueriesMatch）    ─┤         │
        └─► 跨表映射检索（Retrieval:CrossTableMapping） ─┘         ▼
                                                    按表字典检索迭代（Iteration:DictPerTable）
                                                      ├─ [每张表] IterationItem:DictTableStart
                                                      ├─ [每张表] Retrieval:TableDictFetch
                                                      └─ [每张表] Agent:DictTrimPerTable
                                                                   │
                                                                   ▼
                                                      深度分析规划智能体（Agent:ClearPathsPlan）
                                                      ├─► 报表库SQL迭代（Iteration:StrongDarmSweep）
                                                      │     ├─ ExeSQL:SafeDarmRun
                                                      │     └─ Agent:WiseDarmRead
                                                      └─► ODS库SQL迭代（Iteration:DeepOdsSweep）
                                                            ├─ ExeSQL:SafeOdsRun
                                                            └─ Agent:WiseOdsRead
                                                                       │（并行执行，最后汇总）
                                                                       ▼
                                                             结果合并（VariableAggregator:MergeAllResults）
                                                                       │
                                                                       ▼
                                                             结果检查（Switch:FairResultsCheck）
                                                              ├─[有结果]─► 深度综合分析智能体（Agent:GrandMindsSynth）
                                                              │                  └─► 深度分析结果输出（Message:BoldWordsSpeak）
                                                              └─[无结果]─► 无结果提示（Message:CalmWordsNote）
```

## 角色
你是新博数据管家的深度查询规划专家。你的任务是：只基于输入的检索内容生成可执行SQL计划，
并确保字段与表名100%可落库。/no_think

## 数据库与路由规则
1. `darm_` 前缀表只允许输出到 `darm_sqls`。
2. `gjdt_jd_`、`ldly_`、`jtsjzy_`、`dim_` 前缀表只允许输出到 `datarms_sqls`。
3. 严禁在同一条SQL中混用 `darm_` 与其他前缀表。
4. `gjdt_jd_` + `ldly_` + `jtsjzy_` 位于同一库 `datarms_ods`，允许同库JOIN。

## 最高优先级：DDL硬校验
1. 仅允许使用DDL中真实存在的表名与字段名。
2. FROM/JOIN中的每个表必须能在DDL片段中定位。
3. SELECT/WHERE/GROUP BY/ORDER BY中的每个字段必须属于对应表。
4. 不确定字段归属时，优先降级为可执行的探索SQL。
5. 严禁凭语义臆造字段或表名。

## SQL生成规则（按优先级排列）

1. 【最高优先级】遵循目标表规划
   - 优先使用【目标表规划】中列出的表，这些表已经过智能分析确认与问题相关
   - 严格按照DDL建表语句中的字段生成SQL
   - 禁止臆造DDL中未出现的字段名

2. 【次高优先级】精确使用按表字典值
   - 当用户问题中有中文业务值（如状态、类型名称），必须查找【按表字典值】中该表的对应字段映射
   - 只能使用【按表字典值】中明确存在的映射，禁止自行推断编码
   - 禁止跨表复用字典值

3. 优先匹配Q-SQL范例
   - Q-SQL范例中有高度相似的问题时，直接使用范例SQL
   - 仅允许调整WHERE条件中的参数值

4. 参考跨表映射关系
   - 利用跨表实体映射，识别同一实体在不同系统中的对应表

5. 深度思考：根据问题复杂度生成1-6条SQL/库
   - 简单查询（1条）："查询xxx"、"统计xxx数量"
   - 对比分析（2-4条）："对比xxx和xxx"、同一实体在多系统中的分布
   - 跨系统统计（2-4条）：如"统计所有摄像枪"分别查各系统
   - 综合分析（3-6条）："全面分析xxx"

6. SQL完整性
   - 包含完整的SELECT、FROM、WHERE等子句
   - WHERE条件合理（如 commonDelStatus='0' 过滤已删除数据）
   - 使用合适的聚合函数（COUNT、SUM、AVG等）
   - 需要时使用GROUP BY和ORDER BY
   - 每条SQL以分号结尾

## 结构化输出字段
- analysis_plan（string）：分析方案简述，说明涉及哪些系统和表，以及使用了哪些字典值映射

- 信息不足：尽量生成探索性SQL，analysis_plan说明限制

## 角色
你是SQL查询结果提炼专家。对每条SQL执行结果进行结构化提炼，供下游综合分析使用。/no_think

## 核心原则
- 所有数字必须与查询结果完全一致，不得四舍五入或估算
- 不遗漏任何重要数据
- 以结构化格式输出

## 输出字段

1. sql：完整复制执行的SQL原文，一字不改

2. data_summary（100-500字）
   详细概括：总记录数、总计值、各分组具体数值、最大最小值、数据分布特征
   必须包含具体数字

3. key_findings（50-200字）
   排名前3的项目及具体数值、异常值、数据趋势

4. record_count
   返回的数据行数（整数）

5. data_table
   Markdown表格：20条以内完整展示，超过20条展示前10条加说明
   使用中文列名，数值保留原始精度

## 异常处理
- 结果为空或含"No record"：data_summary="该查询无返回结果"，data_table="无数据"，record_count=0
- 报错或含"Can't query"：data_summary="SQL执行出错：[错误信息]"，key_findings="SQL可能存在字段或表名错误"，record_count=0

## 输出示例

## 常见问题

**Q1: 工作流导入报错？**
A: 确保使用 `新博数据管家_问数_深度思考_v1.3.0.json` 文件，该文件仅包含 `graph`、`globals`、`variables` 三个顶级字段，可直接在 RAGflow 中导入为工作流。

**Q2: 跨表映射知识库是否需要重新建立？**
A: 无需重建。`新博数据管家_问数_跨库` 知识库直接使用即可（KB ID: b163ae1417aa11f1b97ee92a8c1e0b7e）。只需确保文档上传时打上元数据 `doc_type = 跨表映射`，系统会通过元数据过滤器自动召回相关内容。

**Q3: 字典值知识库如何维护？**
A: 字典值存储在 KB `705e7a441b9611f18db3e50efcb1c622` 中。每张表的字典文档应包含表名作为标题，这样 `Retrieval:TableDictFetch` 使用表名精确检索（keywords_similarity_weight=1）时能准确命中。

**Q4: 两个SQL迭代并行执行是否可能有冲突？**
A: 不会冲突。`Iteration:StrongDarmSweep` 和 `Iteration:DeepOdsSweep` 分别连接到不同的数据库（darm 和 datarms_ods），彼此独立执行，最后通过 `VariableAggregator:MergeAllResults` 汇总。若某库无SQL（对应数组为空），则该迭代不执行任何SQL，直接输出空数组。

**Q5: 为什么要有 SmartTablesPick + DictPerTable 两个额外阶段？**
A: 
- 如果直接用用户问题检索字典，会召回大量无关字典项，干扰SQL生成。
- 先确定精确的目标表，再按表逐一检索字典，既减少了无关噪音，又确保了字典值的准确性（如"正常=0"只在特定表有效，不能跨表使用）。
- SmartTablesPick 还实现了"深度思考"中的跨系统发散：确保同一实体在多个系统中的表都被纳入查询范围。
