# 知识库检查结论_非预置SQL提示词优化

## 1. 本次检查范围

- `知识库文件/新博数据管家_问数_Q_SQL_v3.0.1`
- `知识库文件/新博数据管家_问数_DB_Description`
- `知识库文件/新博数据管家_问数_DDL`
- `知识库文件/新博数据管家_问数_DICT_v1.0.1`
- `知识库文件/新博数据管家_简称映射.xlsx`

## 2. 检查结论

### 2.1 DB / DDL 基本一致

- 原始表：`20` 张表，`DB / DDL / DICT` 三者一致
- 报表：`14` 张表，`DB / DDL` 一致，但没有 `DICT`
- 机电：`34` 张表，`DB / DDL` 一致，`DICT` 覆盖 `30` 张
- 路运：`32` 张表，`DB / DDL` 一致，`DICT` 覆盖 `30` 张

### 2.2 字典覆盖并不完整

#### 机电缺失 DICT 的表

- `gjdt_jd_base_device_card_supplier_paras`
- `gjdt_jd_base_devops_service_evaluation_explain`
- `gjdt_jd_base_store_inventory_detail`
- `gjdt_jd_monitor_road_cms_list`
- `gjdt_jd_ops_order_fault_device_rel`

#### 路运缺失 DICT 的表

- `ldly_cur_vms_data`
- `ldly_his_vms_data`

### 2.3 Q-SQL 不能被当成绝对真值

Q-SQL 总样例量：

- 原始表：`1042`
- 报表：`378`
- 机电：`931`
- 路运：`1200`

发现明显坏样例：

- 机电：`16 / 931`
- 路运：`106 / 1200`

典型问题是：

- `FROM ... AND ...`，缺少 `WHERE`
- 与当前问题无关的样例被检索出来后容易误导表选择或 SQL 拼接
- 一些样例自带 `LIMIT 100`，但这不是所有问题都应继承的规则

### 2.4 不同知识库对值映射提供的证据强弱不同

- `DICT`：最适合做枚举/状态/类型/方向/标志类条件值映射
- `DB Description`：更适合做业务语义判断、返回字段判断
- `DDL`：最适合做表字段存在性和连接合法性判断
- `Q-SQL`：只适合做“同类问题写法参考”，不适合直接作为字段值真值来源

### 2.5 报表库是特殊情况

- 报表库没有 `DICT`
- 但 `Q-SQL` 中大量稳定使用中文字面值，例如：
  - `settlementStatus = '已结算'`
  - `isDeleted = 'N'`
- 这类场景应优先按“同表同字段稳定样例 + DB/DDL说明”处理，不能强行编码化

### 2.6 简称映射适合放在问题改写阶段

- `新博数据管家_简称映射.xlsx` 是一份小规模实体别名映射表
- 结构是：`正式名称/全称 -> 简称或通称`
- 内容主要是公司或单位名称的简称映射，例如：
  - `华路 -> 广东华路交通科技有限公司`
  - `飞达 -> 广东飞达交通工程有限公司`
  - `长大 -> 保利长大工程有限公司`
- 这类信息最适合在“问题改写”阶段处理，而不是在 SQL 生成阶段处理
- 原因是：
  - 它解决的是检索命中问题，不是 SQL 编码映射问题
  - 一旦在问题改写阶段补全为正式名称，后续 `DB / DDL / Q-SQL` 检索都会更稳定
  - 多轮对话中还可以结合 `sys.history` 把“它 / 这家 / 那个公司的”补足为正式名称

## 3. 对 prompt 的优化策略

### 3.1 表选择器

- 以 `DB / DDL` 为主证据
- `Q-SQL` 降级为辅助证据
- 明确忽略坏样例

### 3.2 字典裁剪器

- 只保留当前问题真正用于过滤、分组、排序或分类展示的字段字典
- 若当前表没有可靠字典，直接返回空数组

### 3.3 SQL 生成器

- 明确证据优先级：`DDL > DB > DICT > Q-SQL`
- 强制坏样例过滤
- 强制值映射必须有证据
- 强制 `LIMIT` 只在“前N / 最新一条”等语义下使用
- 强制软删除过滤必须有同表明确依据
- 强制输出前做语法与证据自检

### 3.4 问题改写器

- 只补足上下文，不改写原始实体值
- 严禁把中文业务值提前改成数据库编码
- 确保检索 query 保留原始编号、日期、公司名、设备名、状态词
- 支持 `sys.history` 进行多轮承接补全
- 对简称映射采用“正式名称（简称）”补全策略，提升检索稳定性
